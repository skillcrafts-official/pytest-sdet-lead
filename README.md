# pytest-sdet-lead
Учебный проект для роста специализации Python SDET

## Проектирование тестов
Любой контрактный тест должен уметь отвечать на вопросы:
1. **Корректность структуры ответа**
   Какие поля есть у успешного профиля? Какие типы данных? Есть ли обязательные поля (`id`, `username` / `email`, `profile` с вложенными полями)?
   _Даже если контракт не описан, мы создаём его прямо в тесте как эталон._
2. **Идентичность данных**
   Запросив профиль пользователя, мы должны сверить, что ответ содержит именно те значения, которые лежат в базе (или сгенерированы фабрикой). Например, `response.data['email'] == user.email`.
3. **Контракт ошибок**
   Для 401, 404, 403 тело ответа должно содержать стандартный DRF-объект с ключом `detail` (или кастомный). Проверяем не только статус, но и сообщение.
4. **Граничные условия**
   Что вернёт запрос `pk=0`, `pk=-1`, `pk=строка`? А если `pk` в UUID-формате (если модель использует UUID)? Мы можем добавить пару кейсов.
5. **Прозрачная параметризация**

### Тест-дизайн для контрактных тестов
Контрактный тест — это не полноценный функциональный тест бизнес-логики. Он отвечает на вопрос: «Соответствует ли ответ ожидаемой структуре и формату?»

#### Какие аспекты проверяются
|Категория|Что конкретно проверяется|Пример (Django REST)|
|--|--|--|
|HTTP-статус|Код ответа|`200`, `201`, `404` (по сценарию)|
|Заголовки|`Content-Type`, возможно `Allow`|`application/json`|
|Структура ответа|Обязательные поля верхнего уровня и вложенных объектов|`id`, `username`, `profile.avatar_url`|
|Типы данных|Тип каждого поля|`id` — `int`, `username` — `str`, `is_active` — `bool`|
|Форматы значений|Если важны: дата (ISO 8601), email, URL|`created_at` — строка вида `"2023-...T...Z"`|
|Обязательность|Какие поля должны присутствовать всегда|`is_active` — обязательно, `last_login` — опционально|
|Граничные/допустимые значения (опционально)|Если контракт жёстко фиксирует enum или диапазон|`status` только `"active"`/`"inactive"`|
